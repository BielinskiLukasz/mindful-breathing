  <name>Fix BUG-01: Countdown Timer Cleanup</name>
  <files>index.html</files>
  <action>
Add explicit countdown timer cleanup in stop() function to prevent accumulation when user interrupts 3s countdown with Stop click. Currently, showCountdown() adds 4 timers to countdownTimers array; cancelCountdown() clears them on normal stop, but rapid Start→Stop during countdown can orphan timers.

**Location:** stop() function around line 1877 (after `if (!isRunning) return;`)

**Implementation:** Add 3-line cleanup before `isRunning = false`:
```javascript
if (isCountingDown) {
  cancelCountdown();
}
```

This ensures any pending countdown timers are cleared before the session stops, preventing memory accumulation on repeated Start/Stop during countdown.

**Why:** Prevents low-grade memory leak from orphaned timeout IDs when user clicks Stop while 3-second countdown is running.

Per BUG-01.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User Input → Duration Validator | Duration slider accepts user values (0-500s); must clamp to safe range (1-300s) |
| App State → localStorage | History entries written to localStorage may exceed quota; must handle gracefully |
| Browser → Wake Lock API | Screen wake lock may be revoked by OS; app must detect and allow graceful degradation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | Duration input (range out-of-bounds) | mitigate | Clamp all input values to DURATION_RANGE.min/max (1-300s) before storing; enforce at change-handler entry point |
| T-03-02 | Denial of Service | localStorage quota exceeded | mitigate | Catch QuotaExceededError by name; display user warning message; history cap (1000 entries max) limits quota risk |
| T-03-03 | Availability | Wake Lock revoked mid-session | accept | Release event listener in place; session continues (screen may sleep, but breathing timer continues); documented as acceptable fallback |

</threat_model>
